# План настройки команды агентов

> Методичка для владельца: как поднять команду без хаоса. Читается один раз при первом запуске. Потом — справочно.

## Что уже есть

После установки заведены рабочие папки и базовые роли:
- coordinator
- marketer
- producer
- copywriter
- designer
- tech

У каждого уже есть базовые файлы:
- `AGENTS.md`
- `IDENTITY.md`
- `SOUL.md`
- `USER.md` (шаблон под заполнение)
- `MEMORY.md`
- `TOOLS.md`

Вывод: переписывать всё с нуля не надо. Основа уже нормальная.

## Главный принцип

Сначала настраиваем не «всех подряд», а единый контекст и правила маршрутизации.

Если этого не сделать, агенты начнут:
- путать проекты между собой
- давать советы не в своей зоне
- хранить дубли и противоречия в памяти

## Как настроить без хаоса

### Этап 1. Зафиксировать общие правила

Нужно определить:
1. кто главный вход в систему — Координатор
2. какие активные проекты есть прямо сейчас
3. кто принимает решения по каждому проекту
4. как помечаем проект в каждой задаче

Рекомендация:
- проект 1 = активный проект, под которым команда работает прямо сейчас
- проект 2 = либо не оформлен (не смешиваем), либо параллельный — но всегда с явной пометкой `[проект2]`
- все новые задачи сначала идут Координатору

### Этап 2. Не трогать лишнее

Пока не лезем глубоко в:
- `AGENTS.md`
- `SOUL.md`
- `IDENTITY.md`

Их меняем только если в работе всплывёт реальная проблема.

Сначала заполняем:
- `USER.md`
- `MEMORY.md`
- `TOOLS.md`

### Этап 3. Что писать в USER.md

В `USER.md` для каждого агента должны быть:
- как к тебе обращаться
- часовой пояс
- формат ответа
- уровень краткости
- как лучше задавать задачи
- кто владелец решений: ты, партнёр / соавтор, или совместно
- правило: не смешивать контекст двух проектов без явной пометки

### Этап 4. Что писать в MEMORY.md

В `MEMORY.md` хранить только устойчивые факты:
- по проекту 1: ниша, ЦА, продукт, ограничения, решения
- по проекту 2: только после того, как появится направление
- по роли агента: что уже решили именно в его зоне

Важно: **не превращать `MEMORY.md` в свалку переписки.** Старое → в `MEMORY_ARCHIVE.md` (есть только у координатора, остальные пусть переносят в конец своего `MEMORY.md` в раздел «Архив»).

### Этап 5. Что писать в TOOLS.md

Сюда складывать локальные рабочие штуки:
- ссылки на папки и материалы
- названия аккаунтов / каналов
- названия проектов
- где лежат референсы, креативы, тексты, техничка
- любые стабильные обозначения, которые агентам полезно знать

## Как лучше взаимодействовать с агентами

### Базовый режим

Лучший режим по умолчанию:
- ты пишешь Координатору
- Координатор решает, кого подключать
- узкий агент работает в своей зоне
- результат возвращается тебе уже собранно

Это самый безопасный сценарий.

### Когда идти напрямую

Напрямую можно писать, если задача узкая:
- **Маркетологу** — оффер, воронка, гипотеза, прогрев
- **Продюсеру** — продукт, цена, формат, запуск
- **Копирайтеру** — пост, письмо, сценарий, лендинг
- **Дизайнеру** — визуал, баннер, обложка, структура экрана
- **Технарю** — код, интеграции, API, деплой

### Правило синхронизации после прямого общения

Если ты плотно поработал с агентом напрямую, должно происходить одно из двух:
- либо агент в конце делает короткую сводку для Координатора,
- либо ты приносишь сюда результаты, а Координатор фиксирует их в общем плане.

Минимум синхронизации после такой сессии:
- что решили
- что изменилось
- что осталось открытым
- что нужно следующему агенту

Без этого знания остаются распылёнными по сессиям, и Координатор потом не видит картину целиком.

### Что важно писать в начале задачи

В идеале каждая задача начинается так:
- проект: `[проект1]` или `[проект2]`
- цель
- желаемый результат
- дедлайн / ориентир
- ограничения

Пример:
> Проект: [проект1]
> Задача: <<сформулировать оффер для Telegram-консультации>>
> Цель: <<5 заявок в неделю>>
> Ограничения: <<без агрессивных обещаний>>

## Что рекомендую прямо сейчас

1. Зафиксировать, что Координатор — единая точка входа
2. Описать `[проект1]` как основной активный контекст
3. Не заполнять пока детали по `[проект2]`, которых ещё нет
4. Прописать единые правила в `USER.md` для всех агентов
5. Потом отдельно заполнить `MEMORY.md` для каждого по своей роли

## Следующий практический шаг

Сначала собрать единый master-brief:
- как тебя называть
- стиль общения
- проект 1: ниша, продукт, ЦА, текущая цель
- проект 2: статус одним предложением
- кто принимает финальные решения
- какой формат ответов тебе удобен

После этого уже можно массово и аккуратно заполнить `USER.md` для всей команды.
