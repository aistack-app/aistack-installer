## Предзаполненные правила (из чужого опыта)

### 1. Сначала бэкап, потом изменения

- `[CORRECTION]` Применил миграцию к боевой базе без бэкапа — миграция
  оказалась с багом, потеряли часть данных
- `[CORRECT]` Любая команда которая меняет состояние (миграция,
  массовое удаление, перезапись файла) → сначала бэкап → потом
  команда. Бэкап в отдельной папке с датой.
- `[RULE]` [safety] Нет бэкапа = нет изменений. Сначала dump
  данных / снимок ФС / git commit, потом исполнение.

### 2. `rm -rf` только после `--dry-run`

- `[CORRECTION]` Выполнил `rm -rf ~/projects/*-temp` — оказалось что
  переменная `~` указывала не туда, удалил не то
- `[CORRECT]` Перед `rm -rf` сначала `ls` или `find` с тем же
  шаблоном — посмотреть что попадает под удаление. Только потом
  `rm`.
- `[RULE]` [safety] Никаких `rm -rf` без предварительного просмотра
  списка. Лучше потратить лишние 30 секунд, чем потерять данные.

### 3. Секреты только через env / vault, не в коде

- `[CORRECTION]` Закоммитил `.env` с API-ключами в публичный
  репозиторий. Через 12 часов кто-то начал использовать ключ.
- `[CORRECT]` Всегда `.gitignore` для `.env`. Если ключ случайно
  попал в коммит — НЕ просто `git rm`, а ротация ключа полностью
  (он уже в истории git и индексирован поисковиками).
- `[RULE]` [security] Секреты только через env-переменные или
  vault (1Password / Vercel env / GitHub secrets). Никогда в коде,
  логах, чатах, скриншотах.

### 4. Непротестированный код != работающий код

- `[CORRECTION]` Написал интеграцию с API, не протестировал,
  отдал клиенту со словами «должно работать». Не работало — клиент
  потерял 2 часа на отладку моего косяка.
- `[CORRECT]` Перед сдачей — запустить хотя бы happy-path. Если не
  могу сам запустить (нет тестовой среды / ключей) — явно говорю
  «не тестировал, нужна проверка на твоей стороне» + даю конкретный
  шаг как проверить.
- `[RULE]` [quality] Сдача = «протестировано + работает» или
  «не протестировано + явно указано».

### 5. Простое решение > сложное

- `[CORRECTION]` На задачу «архивировать старые файлы раз в день»
  предложил кронталог + Docker-контейнер + S3-бакет — клиент в шоке,
  ему нужно было `bash-скрипт + cron`
- `[CORRECT]` Сначала простое (shell-скрипт + cron). Если упрусь в
  ограничения — обоснованно усложняю. Не сразу «энтерпрайз».
- `[RULE]` [architecture] Каждое усложнение должно быть оправдано
  конкретным требованием. «Чтобы потом не переделывать» — не
  оправдание для over-engineering.

### 6. Лимиты и таймауты — обязательны на любых внешних запросах

- `[CORRECTION]` Запустил скрипт который дёргает API без лимита
  частоты — получили rate-limit на час, бизнес-процесс встал
- `[CORRECT]` Любой запрос к внешнему API → таймаут + retry с
  экспоненциальной задержкой + явный лимит requests-per-minute
  (rate limit на нашей стороне)
- `[RULE]` [resilience] Никаких «бесконечных» циклов / запросов.
  Всё с таймаутами и лимитами.
