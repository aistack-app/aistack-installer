## Предзаполненные правила (из чужого опыта)

### 1. Один продукт за раз — не три параллельно

- `[CORRECTION]` Помогал собирать одновременно core + МП + апсейл за
  один спринт. Команда устала, core не отшлифован, МП не запущен.
- `[CORRECT]` Один продукт от идеи до запуска. Только после успешного
  запуска core — следующий (МП, апсейл, и т.д.).
- `[RULE]` [scope] Не запускать N продуктов параллельно. Очередь.

### 2. Scope freeze за 2 недели до запуска

- `[CORRECTION]` За 3 дня до эфира клиент сказал «давай ещё один блок
  добавим в программу». Команда не успела, эфир прошёл слабее.
- `[CORRECT]` За 2 недели до запуска — scope замораживается. Любые
  правки — после запуска и фидбэка реальных клиентов.
- `[RULE]` [process] Дата заморозки scope = дата запуска − 14 дней.
  Нарушение требует обоснования и согласования.

### 3. Не делать продукт «для всех»

- `[CORRECTION]` Программа курса описана для «всех кто хочет улучшить
  жизнь» — конверсия 0.3%, потому что никто не узнаёт себя
- `[CORRECT]` Один портрет (возраст / профессия / ситуация / боль).
  Текст звучит как «это для меня» именно ЭТОЙ группе.
- `[RULE]` [positioning] Узкое позиционирование > широкое. Можно
  расширять после первого успешного запуска.

### 4. Считать unit-эконом до запуска

- `[CORRECTION]` Собрали продукт без проверки экономики — себестоимость
  оказалась 60% от цены, бизнес не масштабируется
- `[CORRECT]` ДО старта работы над продуктом: считаем себестоимость
  (себестоимость материалов + время команды + комиссии платёжки),
  маржу, минимальное количество продаж для безубыточности
- `[RULE]` [economics] Маржа core-продукта должна быть минимум 50%
  с учётом всех комиссий. Иначе — менять цену или модель.

### 5. Не игнорировать команду — она знает что не работает

- `[CORRECTION]` Продюсер настоял на сжатом сроке запуска. Копирайтер
  сказал «не успею» — настоял. Тексты вышли сырыми, продаж меньше.
- `[CORRECT]` Если узкий специалист (Копирайтер / Дизайнер / Технарь)
  говорит «не успею» — переспросить «что мешает?» Если ограничение
  реальное — двигать дедлайн, не сжимать качество.
- `[RULE]` [team] Узкий специалист в своей зоне знает лучше. Спор
  допустим, игнорирование — нет.
