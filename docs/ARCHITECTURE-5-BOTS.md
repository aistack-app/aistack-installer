# AIStack · Сборка «Малый бизнес» — 5 ботов-отделов (v4)

Архитектура отдельной сборки под малый локальный бизнес (фитнес-студии,
бьюти-салоны, кофейни/гастро). Не заменяет сборку из 8 ролей — живёт рядом
как пресет `smallbiz`.

**Решения владельца (зафиксированы 2026-06-12):**
- Проактивность — через встроенный **HEARTBEAT** OpenClaw (не отдельный
  Python-воркер: ещё один процесс = ещё одна точка отказа).
- «Хозяин-секретарь» — **async**: передал задачу отделу → сразу отвечает
  «передал в <отдел>, вернусь с результатом», не держит владельца у экрана.
- hermes-agent остаётся в установке (решение 2026-06-11).

---

## 1. Почему 5 ботов, а не 8 ролей и не 1 супер-бот

Проверено на предыдущих итерациях продукта:
- **v1, 6–8 широких ролей** — владелец малого бизнеса не понимает, кому
  писать («маркетолог или контентщик?»).
- **v2, 15 узких ролей** — перегруз: 15 чатов в Telegram никто не открывает.
- **v3, 1 бот + LLM-роутер** — сложно, дорого, непредсказуемо роутит.
- **v4, 5 ботов-отделов** — как маленькая компания: понятно, кому писать,
  а если написал не туда — Хозяин сам передаст.

## 2. Состав

| Бот | Отдел | Зона | Username-паттерн |
|---|---|---|---|
| 🎙️ Голос | Клиенты | приём обращений, запись, возврат «спящих», отзывы/негатив | `voice_<studio>_bot` |
| 🖋️ Перо | Контент | посты, сторис, карусели, сценарии Reels, тон бренда | `pero_<studio>_bot` |
| 📊 Рост | Маркетинг | конкуренты, реклама, аналитика, офферы | `rost_<studio>_bot` |
| ⏰ Часы | Операционка | утренний бриф, дедлайны, график мастеров | `chasy_<studio>_bot` |
| 🎩 Хозяин | Координатор | общая картина, маршрутизация, эскалации | `boss_<studio>_bot` |

Плюс **private-канал «Команда <Студия>»** — лента команды: итоги дня,
важные события, еженедельный отчёт. Владелец читает канал, пишет ботам.

## 3. Навыки внутри бота — SKILLS/*.md (без LLM-роутера)

У каждого бота в воркспейсе папка `SKILLS/` с файлами-навыками:

```
workspace-voice/
  SOUL.md          # характер отдела + правила маршрутизации между навыками
  IDENTITY.md      # роль, границы, эскалации
  SKILLS/
    zapis.md       # запись клиента (день/время/мастер → подтверждение)
    vozvrat.md     # «спящие» клиенты: когда и что писать
    otzyvy.md      # работа с отзывами, протокол негатива
  MEMORY.md        # долговременная память (встроенная файловая OpenClaw)
  HEARTBEAT.md     # проактивные задачи отдела (см. §5)
  TOOLS.md
```

Выбор навыка — обычное чтение контекста моделью по правилам из SOUL.md
(«про запись/перенос — SKILLS/zapis.md», «про отзыв — SKILLS/otzyvy.md»).
Никакого классификатора: модель и так это умеет, правила делают выбор
предсказуемым.

## 4. Координация — event-bus файлом

`~/.aistack/events.log` — JSONL, append-only, один файл на установку.

```json
{"id":"<uuid>","ts":"2026-06-12T08:00:12+03:00","from":"chasy","type":"morning_brief","payload":{...}}
{"id":"<uuid>","ts":"...","from":"khozyain","type":"task_for_department","dept":"pero","task":"..."}
{"id":"<uuid>","ts":"...","from":"pero","type":"task_completed","ref":"<uuid задачи>","result":"..."}
{"id":"<uuid>","ts":"...","from":"voice","type":"bad_review","payload":{"source":"2gis","text":"..."}}
```

Правила:
- **Пишут все, читает каждый своё** (фильтр по `dept`/`type` при чтении).
- **Дедуп по `id`**: обработанные id агент помнит в MEMORY.md → событие
  не обрабатывается дважды (страховка от повторов владельца и ретраев).
- Файл append-only, агенты пишут через стандартные файловые инструменты
  OpenClaw — отдельный демон не нужен.
- Ротация: Часы на еженедельном тике переносят события старше 30 дней
  в `events-archive-YYYY-MM.log`.

## 5. Проактивность — HEARTBEAT (решение №1)

Используем встроенный механизм OpenClaw: периодический тик будит агента,
агент читает свой `HEARTBEAT.md` и решает, есть ли работа.

| Агент | Интервал | Что делает на тике |
|---|---|---|
| Часы | 30 мин | 08:00±тик — утренний бриф в канал; проверка дедлайнов («через 2 дня — событие deadline_in_2_days»); пятница 18:00 — еженедельный отчёт |
| Хозяин | 15 мин | читает event-bus: новые task_completed → доставить владельцу результат; bad_review → эскалация; собирает картину дня |
| Голос | 60 мин | «спящие» клиенты по списку из MEMORY; напоминания о записях завтра |
| Перо | выкл (v1) | работает только по запросу |
| Рост | выкл (v1) | работает только по запросу |

Точность тайминга = интервал тика (бриф «в 08:00» приходит 08:00–08:30) —
для малого бизнеса достаточно, фиксируем в доке продукта честно.

## 6. «Хозяин-секретарь» — async (решение №2)

Сценарий: владелец пишет Хозяину «надо пост про новую услугу» (не туда —
это к Перу).

1. Хозяин СРАЗУ отвечает: «Передал Перу, вернусь с результатом» (< 5 сек).
2. Пишет в event-bus `task_for_department` для pero.
3. Перо на ближайшем тике/обращении видит задачу, делает, пишет
   `task_completed` с результатом.
4. Хозяин на своём тике видит completed → присылает владельцу результат
   в тот же чат: «Перо подготовил: …».

Никаких «печатает…» по три минуты. Правило зашивается в SOUL.md Хозяина.

## 7. Wizard (донастройка при установке, цель — 5–10 минут)

Пресет `smallbiz` меняет Stage 6 установщика:

1. Название студии/бизнеса (используется в именах и текстах).
2. **5 токенов** от BotFather — по одному на отдел; установщик подсказывает
   имя для каждого (`voice_<studio>_bot`…). Подключение с паузами 2–3 сек
   (Telegram банит серии — уже реализовано в register_bots).
3. Telegram ID владельца (@userinfobot) — allowlist на все 5 ботов
   (уже реализовано: OWNER_TG_ID).
4. ID private-канала: владелец создаёт канал, добавляет Хозяина админом,
   пересылает любой пост канала боту @JsonDumpBot → берёт `chat.id`.
   Пишется в конфиг Хозяина (`channel-config.json` в воркспейсе).
5. API-ключ модели (как сейчас).
